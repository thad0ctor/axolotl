# Vendored: z-lab/sparselora

This directory contains a vendored copy of the [`sparselora`](https://github.com/z-lab/sparselora) Python package — bundled into Axolotl rather than installed via pip, with a small set of local patches documented under [Local edits](#local-edits) below. It is otherwise upstream's code at the pinned commit.

- **Upstream:** https://github.com/z-lab/sparselora
- **Commit:** `a2fd69de93b1168080346ec113c99501f0bb58b1`
- **License:** MIT (see `LICENSE`)
- **Paper:** SparseLoRA: Accelerating LLM Fine-Tuning with Contextual Sparsity (ICML 2025), arXiv:2506.16500

## What was copied

The `sparselora/` package only (`api.py`, `callback.py`, `config.py`, `__init__.py`, and `modules/`). The upstream `experiments/` directory (training/eval harness and its extra dependencies) was **not** vendored.

## Local edits

These changes were applied to the copied sources:

- Absolute intra-package imports (`from sparselora.… import …`) were rewritten as relative imports (`from .… import …`) so the package works under `axolotl.integrations.sparselora._vendor.sparselora`. Affected: `__init__.py`, `callback.py`, `modules/svd.py`. The example in `modules/registry.py`'s docstring was left as-is.
- `modules/registry.py`: the fused projection names `qkv_proj` and `gate_up_proj` (Phi3-style) were added to `_OUT_PROJS`. `get_sparsity_mode` keys the sparsity direction off the projection's leaf name; a fused `qkv_proj`/`gate_up_proj` base layer is output-sparse in exactly the same way as the separate `q_proj`/`k_proj`/`v_proj` and `gate_proj`/`up_proj` it concatenates, so its base layer must wrap as `out_scatter`. Without this it would default to `None` (dense) and the Phi3 sparse modules' fused, combined-index projection would silently stay dense. Upstream derived the projection name via `sub_name.split(".")[0]` (the *first* path segment); this was changed to the actual leaf (`parts[-1]`, or `parts[-2]` for a `.base_layer` child) so classification is correct regardless of how deeply the `layer_sparsity` schedule keys the module — the plugin keys on `.self_attn`/`.mlp` (making `sub_name` projection-relative, where the two coincide), but a layer-level key would otherwise misclassify every projection as dense. The arch-neutral fused sparse modules live (un-vendored) in `axolotl.integrations.sparselora.arch_wiring`.
- `modules/svd.py`: a float-dtype guard (`_float_dtype`) so predictor factors load correctly when the base is a bitsandbytes 4-bit (QLoRA) weight, whose storage dtype is `uint8`. Upstream inferred the cast dtype directly from `base.weight.dtype`, which would cast the float SVD factors to `uint8` and corrupt them.
- `api.py`: non-nesting mask binding in `_patch_forward`/`_patch_generate`. Upstream did `m.forward = partial(m.forward, masks=...)` on every forward, which nests a fresh `functools.partial` around the previous one each step and keeps every step's `masks` tensor alive for the whole run (a slow leak + growing call depth). The edit captures each module's original bound `forward` once (`m._sparse_base_forward`) and rebinds `masks` onto that, so each step replaces the previous wrapper.
- `modules/svd.py`: `_load_tensors` was split so the `load_file` + `.to(device, dtype)` happens inside an `@functools.lru_cache`-d helper keyed on `(path, str(device), dtype)`. Upstream re-read and re-moved the entire factors safetensors on **every** `create_mlp_predictor` / `create_attn_predictor` call — once per sparsified module — making apply-time O(layers²) in file I/O (the whole factors file read ~60× for a typical run). The create_* functions only *read* the returned dict (`torch.stack` / `load_state_dict`, both of which copy), so sharing one cached dict across all modules in an apply is safe. (Addresses CodeRabbit finding #5.)
- `modules/linear.py`: `lora_forward` now skips the `indices` positional when `indices is None`, calling `self.base_layer(x, *args, **kwargs)` and `lora_B(lora_A(x_in, *args, **kwargs), *args, **kwargs)` instead of threading `None` through. In a valid plugin run the plugin's `orphan_lora` validation guarantees every patched base is a `SparseLinear` (which accepts the extra arg), but the `peft…lora.layer.Linear.forward` patch is process-global; this defensive guard keeps a plain `nn.Linear` base from choking on an extra positional. The sparse path (`indices is not None`) is byte-for-byte unchanged. (Addresses CodeRabbit defensive finding #4.)
- `modules/svd.py`: the factors `lru_cache` is bounded to `maxsize=2` (was `maxsize=None`). The cached dict holds tensors already moved to the compute device, so an unbounded cache pinned a full factors copy in VRAM for every distinct `(path, device, dtype)` and never evicted — a leak across multiple applies in one process. A small bound keeps the within-apply "one load shared by all modules" win while letting old device tensors free. (Addresses CodeRabbit #62 finding: global tensor cache can pin VRAM.)
- `api.py`: `_compute_output_token_mask` now derives the output-token mask from `labels` alone instead of taking `input_ids`. Upstream allocated the mask with `torch.zeros_like(input_ids)`, which crashed when `input_ids` was passed positionally (so `kwargs.get("input_ids")` was `None`). `labels` is always present when the mask is computed and has the same shape, so the mask is built from `labels` directly. The batch-wide contiguous `[left:right]` span is preserved (the `fast_tokens` fast path depends on a single contiguous output block). (Addresses CodeRabbit #62 finding: build the mask without needing positional `input_ids`.)
- `api.py`: the `Trainer.__init__` auto-injection now reads the SparseLoRA schedule from the trainer's own model (`model._sparselora_schedule`, stamped on by `apply_sparselora`) instead of a `SparseLoRAConfig` captured in a closure when the patch was first installed. Upstream closed over the *first* model's `start_step`/`end_step`, so a **second** `transformers.Trainer` built in the same process would auto-inject the first model's schedule. `_patch_trainer()` therefore no longer takes a config argument. The Axolotl plugin always adds its own `SparseLoRACallback` in `plugin._apply`, so the single-model path is unaffected (the auto-inject only matters for a hypothetical second model). (Addresses CodeRabbit finding #1.)

Each vendored file carries a one-line provenance header. **Do not hand-edit these files** — to update, re-copy from upstream at a new pinned commit and re-apply the import rewrite.

## Note on SVD predictor factors

Upstream `modules/svd.py` only *loads* pre-computed predictor factors from a `model.safetensors` (z-lab publishes these for Llama-2/3 only). The offline SVD computation that produces those factors is **not** in the upstream repo. Axolotl's plugin adds that step (`axolotl.integrations.sparselora.factors`) so predictors can be derived from any base model's weights, writing a `model.safetensors` in the exact key layout the vendored loader expects.
